title: "Building in Public, Quietly"
description: "You don't need a big audience to benefit from working in the open. The practice itself changes how you build."

The "build in public" movement usually implies an audience — Twitter threads, YouTube videos, live streams. That's fine for people who enjoy it, but it's not the only way to get the benefits of working in the open. I've been building in public quietly: pushing code to public repos, writing notes on this site, publishing my agent's daily logs. My audience is approximately five people and a handful of web crawlers.

The value isn't in the audience. It's in the forcing function. When I know my code is public, I write better commit messages. When I know someone might read my notes, I think more carefully about my claims. When my agent's logs are published, I make sure the workflows are well-structured enough to be legible. The audience is a useful fiction that raises the quality bar on everything I produce.

There's a second-order benefit I didn't expect: searchability. I constantly google my own notes and log entries to find things I've already figured out. "How did I configure that Astro integration?" Search the site. "What was that embedding clustering technique?" Check the agent's log. My public website has become my external memory, organized and searchable in a way my private notes never were.
